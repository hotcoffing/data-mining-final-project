## 1. Constitution (불변 원칙)

본 문서의 Constitution은 프로젝트 전 기간 동안 **변경하지 않는** 원칙이다. `techspec.md`, `tasks.md`, 구현 코드는 이 원칙을 위반할 수 없다.

### 1.1 목적 원칙

- 사용자가 **집(출발지)에서 언제 출발하면 목적지 인근에서 따릉이를 탈 수 있는지**를 데이터·머신러닝·지도 API로 추정한다.
- 최종 사용자 경험은 **Jupyter Notebook 한 곳**에서 입력부터 예측·시각화까지 완결된다.
- 상업 서비스가 아닌 **학습·연구 목적**이므로, 재현 가능성과 방법론 보고가 속도·규모보다 우선한다.

### 1.2 품질 기준

- 필수 ML 기법 4가지를 **모두** 구현하고 Notebook·문서에 **성능 비교까지** 보고한다.
  1. 군집화 (시작/도착 지점 → 약 200m 구역)
  2. 선형 회귀 (가용성·출발 시각 관련 예측)
  3. Random Forest (동일 목표, 비선형 비교)
  4. Random Forest 하이퍼파라미터 튜닝 (튜닝 전·후 성능 비교)
- 카카오맵 API로 **경로·소요시간**을 계산하고 예측 파이프라인에 반영한다.
- `requirements.txt`만으로 환경을 재현할 수 있어야 한다.
- 단계별 산출물(`prd.md` → `techspec.md` → `tasks.md` → 구현)은 **검수 승인 후**에만 다음 단계로 진행한다.

### 1.3 UTF-8 저장 원칙 (필수)

- 프로젝트에서 **새로 생성·수정하는 모든 텍스트 파일**은 **UTF-8**로 저장한다.
- **적용 대상**: `prd.md`, `techspec.md`, `tasks.md`, `docs/**/*.md`, `README.md`, `*.py`, `*.ipynb`, `requirements.txt`, `.env.example` 등
- **금지**: cp949, EUC-KR, ANSI 등 비 UTF-8로 산출물 저장
- **읽기 예외**: `data/` 원본 CSV·XLSX는 `cp949`, `utf-8-sig` 등으로 **읽기만** 허용. 읽기 인코딩은 `techspec.md`에 명시한다.
- 생성 직후 한글 포함 파일은 Cursor에서 깨짐 없음을 확인한 뒤 완료로 간주한다.

### 1.4 금지 사항

- 검수 승인 없이 다음 단계 문서·구현을 진행하지 않는다.
- 원본 대용량 CSV를 Notebook에서 **전량 로드**하지 않는다. (샘플링·월별 분할·사전 집계 필수)
- API 키를 코드·Git에 커밋하지 않는다. (`.env` 사용, `.gitignore`에 `.env` 포함)
- 월별 이용건수만 있는 데이터를 **시간대별 가용성 예측의 주 데이터**로 사용하지 않는다.
- 상업 배포·실시간 운영을 전제로 한 과도한 인프라(서버, DB, 스케줄러)를 도입하지 않는다.

### 1.5 세션·에이전트 운영

- 메인 세션은 `prd` → `techspec` → `tasks` → 통합·검수의 **줄기**를 담당한다.
- 독립 하위 작업은 서브에이전트로 수행하며, 완료 시 `docs/`에 **UTF-8 마크다운 보고서**를 남긴다.
- 메인 세션은 서브에이전트 보고서를 읽고 `tasks.md` AC 및 Notebook에 반영한다.

## 2. 문제 정의

### 2.1 사용자 문제

서울 시민이 집에서 출발해 따릉이로 이동하려 할 때, **가까운 대여소에 자전거가 있을 시간**을 모르면 헛걸음하거나 대안 교통을 써야 한다. 본 프로젝트는 다음 질문에 답한다.

> **「지금 위치(집)에서 목적지까지 가려면, 몇 시에 출발해야 출발지 인근 대여소에서 따릉이를 탈 수 있을까?」**

### 2.2 핵심 개념

| 개념 | 설명 |
|------|------|
| 구역(Cluster) | 대여소 좌표를 군집화해 형성한 **최대 약 200m** 범주의 공간 단위. 사용자 입력 주소·좌표를 구역에 매핑 |
| 가용성 | 특정 구역·시각에 인근 대여소 **거치대수량 > 0** (또는 임계값 이상) 여부 |
| 소요시간 | 카카오맵 API(도보·자전거 등 정책은 techspec에서 확정)로 계산한 **출발지 → 최근접 대여소 → 목적지** 또는 **출발지 → 목적지** 소요시간 |
| 출발 권장 시각 | 목표 도착(또는 대여) 시각에서 소요시간을 역산하고, 그 시각의 **예측 가용성**이 충족되는 최소 출발 시각 |

### 2.3 예측 목표 (초안)

- **회귀**: 출발 시각·요일·구역·계절 등 특성으로 **거치대수량**(또는 구역 내 평균 가용 대수) 예측
- **의사결정**: 예측 가용 대수(또는 가용 확률) ≥ 사용자 임계값(기본 1대)을 만족하는 **가장 늦은 출발 시각** 제시
- **비교**: 선형 회귀 vs Random Forest vs 튜닝된 Random Forest의 MAE/RMSE/R² 등을 Notebook에 보고

### 2.4 범위 (In / Out)

**In Scope**

- 서울시 공공자전거(따릉이) 공개 데이터 기반 오프라인 분석
- Notebook 내 `input()` 또는 ipywidgets 사용자 입력
- 카카오맵 REST API (무료 티어) 연동
- 군집화 결과 지도 시각화 (folium 등)

**Out of Scope**

- 실시간 운영·배포·모바일 앱
- 전국/타 지역 확장
- 날씨·이벤트 등 외부 실시간 API (시간 부족 시 제외, techspec에서 확정)

### 4.5 A–B 조인 및 C(대여이력) 재검토

`data_2401.csv` 샘플(약 8만 행)과 `25.12월` 대여소 XLSX 간 **대여소번호 정규화 후** 매칭률을 확인했다. 상세 수치는 `docs/_join_key_analysis.json`에 기록.

| 지표 | 결과 |
|------|------|
| A 고유 대여소번호 (샘플) | 3,069 |
| B 고유 대여소번호 | 2,780 |
| 번호 기준 A→B 매칭률 | **87.4%** |
| A에만 존재 (미매칭, 샘플) | 386 |
| 미매칭 중 **대여소명**으로 B 매칭 | 0 |
| 미매칭 중 C(2501) 번호로 추가 발견 | **9** (386 중) |
| C 경유 이름→번호→B 연결 | 0 |

**결론**

- A와 B는 **대여소번호 정규화**(앞자리 0 제거 등)만으로 **대부분(≈87%)** 조인 가능하다.
- 미매칭 약 13%는 2024년에 존재하나 2025.12 마스터에 없는 **폐쇄·구형 대여소**로 추정된다. 대여소명·C 대여이력으로 추가 연결 가능한 건은 **극소(9건)** 에 불과하다.
- C(월 300MB급)를 v1에 포함할 **비용 대비 효과가 없다**. v1에서는 **B와 조인된 A 행만 사용**하고 미매칭 행은 제외한다.
- B 마스터 CSV(`대여소_ID`, 위도, 경도)는 techspec에서 **보조 룩업**(필요 시)으로만 검토하며, v1 필수 데이터는 **25.12월 XLSX**로 한정한다.

## 5. 사용자 시나리오 (Notebook UAT 관점)

### 5.1 기본 시나리오

1. 사용자가 Notebook을 실행하고 의존성을 설치한다 (`requirements.txt`).
2. `.env`에 카카오 REST API 키를 설정한다.
3. Notebook에서 **출발지 주소(또는 좌표)**, **목적지 주소(또는 좌표)**, **희망 도착 시각**(또는 여유 시간)을 입력한다.
4. 시스템이 주소를 지오코딩하고, 출발·도착지를 **군집 구역**에 매핑한다.
5. 카카오맵 API로 **소요시간**을 계산한다.
6. 회귀·Random Forest 모델이 **출발 시각별 예측 가용 대수**를 산출한다.
7. 「**권장 출발 시각**」과 모델별 성능 요약, 구역 지도가 출력된다.

### 5.2 예시 입력 (UAT용 초안)

| 항목 | 예시 값 |
