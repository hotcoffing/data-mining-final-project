| 항목 | 내용 |
|------|------|

| 평가 범위 | `data/`, `src/`, `scripts/`, `notebooks/` |

## 1. 개요

본 보고서는 구현 산출물만을 기준으로, **관찰된 이슈 → 원인 → 프로젝트 한계**를 대응 관계로 기술한다. 각 한계는 “버그”와 “의도된 단순화”를 구분하며, 후자는 기말 과제 범위·데이터·연산 자원 제약에서 비롯된 설계 선택으로 해석한다.

### 전체 한계의 축 (요약)

| 축 | 내용 |
|----|------|
| **시간** | 2024년 과거 패턴으로 미래(실행 시점) 출발 시각을 추정 |
| **공간** | 200m 구역·구역 평균 가용 대수로 개별 대여소를 대표 |
| **데이터** | 3개월·샘플 학습, A↔B 조인 미매칭 제외 |
| **모델** | 시간·구역 위주 특성, 외부 요인(날씨·이벤트) 미반영 |
| **서비스** | 실시간 API 가용이 아닌 오프라인 학습·배치 예측 |

## 2. 데이터 수집 (`data/`)

### 2-1. 다수 수집·소수 사용

| 항목 | 내용 |
|------|------|
| **이슈** | `data/`에 2024 가용량, 2025 대여소, 대여이력, 월별 이용, 2019~2021 구 데이터 등이 있으나, 파이프라인은 **2024 가용 3개월(2401/2404/2407) + 2025.12 대여소 XLSX**만 사용한다. |
| **설명** | 대여이력(월 ~300MB)은 트립 단위라 “특정 시각 거치대수” 스냅샷이 아니며, 월별 이용정보는 시간 해상도가 부족하다. 코드상 `AVAILABILITY_MONTHS`, `station_xlsx()`가 이 조합만 가리킨다. |
| **연관 한계** | **데이터 범위 한계** ? 계절·요일 패턴은 1·4·7월 겨울·봄·여름 대표 샘플에 국한되며, 연말·휴일·이벤트 등은 학습·검증에 충분히 반영되지 않을 수 있다. |
| **영향** | Notebook·예측 결과는 “서울 전체·연중”이 아니라 **선택된 3개월의 통계적 평균**에 가깝다. |

### 2-2. 시점 불일치 (2024 가용 + 2025 좌표)

| 항목 | 내용 |
|------|------|
| **이슈** | 가용 수량은 2024년, 좌표·대여소 마스터는 2025.12 기준이다. `norm_station_no` 후 inner join으로 **매칭되지 않는 대여소 행은 ETL에서 제외**된다. |
| **설명** | `etl._process_chunk`는 `station_keys`에 없는 `station_no_norm`을 필터링한다. 2024에만 있거나 2025에 폐쇄된 대여소는 분석 대상에서 빠진다. |
| **연관 한계** | **공간 커버리지 한계** ? 조인 실패 구간의 가용 패턴은 모델이 학습·예측하지 못한다. 구역 평균(`bike_count_mean`)은 남은 대여소만으로 집계된다. |
| **영향** | 특히 신설·폐쇄가 많은 구역에서 예측과 실제 체감 가용성의 괴리가 커질 수 있다. |

### 2-3. 원본 Git 미포함

| 항목 | 내용 |
|------|------|
| **이슈** | `.gitignore`에 `*.csv`, `*.xlsx`가 있어 원본 데이터는 저장소에 없고, `data/processed/` 캐시에 의존한다. |
| **설명** | 제출·재현 시 수동으로 공공데이터를 받아 동일 경로에 두거나, 이미 생성된 processed 파일이 있어야 한다. |
| **연관 한계** | **재현성 한계** ? 환경은 `requirements.txt`로 맞출 수 있으나, **동일 원본·동일 ETL**을 보장하려면 데이터 수집 단계가 별도 전제가 된다. |

## 3. ETL·전처리 (`src/etl.py`, `src/stations.py`)

### 3-1. 구역 단위 집계로 정보 손실

| 항목 | 내용 |
|------|------|
| **이슈** | `aggregate_zone_hour`가 `(date, hour, cluster_id)` 단위로 `bike_count_mean`, `bike_count_max`를 산출하고, ML 타깃은 **평균(`bike_count_mean`)** 이다. |
| **설명** | 같은 구역 안에서도 대여소마다 거치 대수가 다르다. 평균은 “구역 어딘가에 자전거가 있다”는 느슨한 신호이며, 사용자가 도착하는 **최근접 대여소**와 불일치할 수 있다. |
| **연관 한계** | **공간 해상도 한계** ? 예측은 구역 패턴, 의사결정은 `nearest_station` 한 곳에 의존한다. 구역 평균 3대여도 최근접 대여소가 0대일 수 있다(코드는 구역 모델만 사용). |
| **영향** | `predict_origin_rental`의 가용 판정이 **실제 대여 가능성을 과대·과소 추정**할 수 있다. |

### 3-2. 청크 ETL과 메모리·완전성

| 항목 | 내용 |
|------|------|
| **이슈** | 월별 `chunksize=100_000` 처리 후 `pd.concat`으로 합친 뒤 `availability_station_hour.csv` 전체를 저장한다. 3개월 합산 시 **수백만 행** 규모가 된다. |
| **설명** | 청크 읽기는 “한 번에 전월 로드” 문제는 피하지만, 최종 concat·집계 단계에서는 여전히 대용량 메모리·디스크가 필요하다. |
| **연관 한계** | **연산 자원 한계** ? 이후 ML은 전량이 아닌 샘플(`ML_SAMPLE_ROWS=80_000`)로 학습한다. ETL은 풍부한데 학습은 축소된 대표 표본에 의존한다. |

### 3-3. 컬럼 위치 기반 파싱

| 항목 | 내용 |
|------|------|
| **이슈** | `_process_chunk`가 `iloc[:, 0]`~`iloc[:, 4]`로 컬럼을 읽는다. |
| **설명** | 공공 CSV 스키마가 바뀌면 조용히 잘못 매핑될 수 있다. 현재 파일 형식에 강하게 결합되어 있다. |
| **연관 한계** | **유지보수·이식성 한계** ? 데이터 형식 변경 시 ETL 전면 수정이 필요하다. |

## 4. 군집화 (`src/clustering.py`)

### 4-1. DBSCAN `min_samples=1`과 200m 해석

| 항목 | 내용 |
|------|------|
| **이슈** | `eps=200m`, `min_samples=1`이면 이론상 인접 대여소가 연쇄적으로 묶여 **구역 직경이 200m를 넘는 클러스터**가 생길 수 있다. 리포트상 직경 250m 초과 구역이 다수 존재한다. |
| **설명** | DBSCAN의 eps는 “이웃 반경”이지 “클러스터 최대 직경”이 아니다. `min_samples=1`은 단일 대여소도 클러스터가 되게 하여, 평균 **1.6대여소/구역**처럼 대부분이 사실상 단일 대여소 구역이 된다. |
| **연관 한계** | **공간 단위 한계** ? “약 200m 구역”은 목표이지, 모든 구역이 엄밀히 200m 이내는 아니다. 출발지→구역 매핑(`assign_cluster`)은 **구역 중심과의 거리**로 하며, 200m 밖이면 경고만 출력한다. |
| **영향** | 구역 ID가 곧 “도보권”이 아니며, 가용성은 인근이 아닌 **같은 cluster_id를 공유하는 먼 대여소** 통계에 섞일 수 있다. |

### 4-2. Notebook 지도 시각화의 단순화

| 항목 | 내용 |
|------|------|
| **이슈** | Notebook §3은 대여소 300개 샘플 점만 표시하고, `cluster_id`별 색·경계·200m 원은 그리지 않는다. |
| **설명** | `cluster_diameter_stats`는 코드에 있으나 Notebook EDA 파이프라인에 연결되지 않았다. |
| **연관 한계** | **검증 가시성 한계** ? 군집 품질 이슈(직경 이상치)를 Notebook만 보고는 파악하기 어렵다. |

## 5. 머신러닝 (`src/ml_models.py`, `scripts/train_models.py`)

### 5-1. `cluster_id` 수치 처리와 선형 회귀 실패

| 항목 | 내용 |
|------|------|
| **이슈** | `NUM_FEATURES`에 `cluster_id`가 포함되고 `StandardScaler`로 스케일링된다. 범주형 구역(1,706개)을 **순서 있는 숫자**처럼 다룬다. |
| **설명** | 선형 회귀는 `cluster_id` 100과 101이 “가까운 구역”처럼 선형 관계를 가정한다. 실제로는 임의 라벨이다. 결과적으로 **R² ? 0.009** 수준의 설명력에 그친다. |
| **연관 한계** | **모델 가정 한계** ? 선형 회귀는 본 과제에서 **베이스라인·대비군** 역할에 가깝고, 구역 효과를 제대로 넣으려면 one-hot·target encoding 등이 필요하다. RF는 트리 분할로 우연히 구역을 나누어 **R² ? 0.44~0.48**을 낸다. |
| **영향** | Notebook에서 “Linear vs RF” 비교 시, 차이는 **알고리즘만이 아니라 특성 처리 적합성**에서도 크게 난다. |

### 5-2. 학습 샘플 80,000행

| 항목 | 내용 |
|------|------|
| **이슈** | `train_models.build_sample()`이 `ml_train.csv`에서 최대 60만 행 읽은 뒤 **8만 행 무작위 샘플**을 저장·학습한다. Notebook도 `ml_train_sample.csv`가 있으면 우선 사용한다. |
| **설명** | `zone_hour` 전체는 약 267만 행이나, `ML_MAX_ROWS=200_000`·`ML_SAMPLE_ROWS=80_000`로 의도적 축소했다. `train_models()` 내부에서 또 `max_rows` 초과 시 `sample()`할 수 있다. |
| **연관 한계** | **통계 대표성 한계** ? 희귀 시간대·구역 조합이 샘플에서 빠지면 테스트·실사용 예측이 불안정해진다. 성능 지표는 “8만 행 기준 hold-out”에 묶인다. |
| **영향** | 동일 입력에서 Linear 9.3대 vs RF 0.8대처럼 **모델 간 예측 편차**가 클 때, 샘플·구역 커버리지 부족이 한 원인이 될 수 있다. |

### 5-3. 제한된 특성 집합

| 항목 | 내용 |
|------|------|
| **이슈** | 특성은 `hour`, `dow`, `is_weekend`, `month`, `season`, `cluster_id`뿐이다. `qr_slots`(거치대 용량), 날씨, 공휴일, 대여소별 이력 등은 미사용이다. |
| **설명** | ETL·stations에 `qr_slots`가 로드되나 `ml_train`까지 전달되지 않는다. |
| **연관 한계** | **예측 변수 한계** ? 같은 시각·구역이라도 **용량·수요 충격**에 따른 변동은 설명하지 못해 R² 상한이 낮다(현재 RF도 분산의 약 절반만 설명). |
| **영향** | “몇 시쯤이냐”는 맞을 수 있어도 “그날 그 구역이 특별히 비었는지”는 맞추기 어렵다. |

### 5-4. 시계열 hold-out과 GridSearch `cv=2`

| 항목 | 내용 |
|------|------|
| **이슈** | `time_split`은 월별 마지막 7일을 test로 두어 **미래 누수 방지** 의도는 좋다. 다만 GridSearchCV는 `cv=2`, RF 기본 `n_estimators=50`으로 연산을 줄였다. |
| **설명** | 교차검증 fold가 적으면 하이퍼파라미터 추정 분산이 크다. 튜닝 전·후 차이(MAE 5.17→5.02)는 **소폭 개선**에 그친다. |
| **연관 한계** | **튜닝 신뢰도 한계** ? “튜닝으로 성능이 크게 오른다”기보다, RF family 내 미세 조정 수준으로 해석하는 것이 타당하다. |

## 6. 예측·API (`src/predict.py`, `src/kakao_map.py`)

### 6-1. 목적지·자전거 구간 미연결 (핵심 시나리오 단순화)

| 항목 | 내용 |
|------|------|
| **이슈** | `predict_origin_rental`은 **출발지 + 할당 가능 시간(분) + 기준 시각**만 받는다. `kakao_map.get_duration(..., "bike")`는 구현되어 있으나 **예측 파이프라인에서 호출되지 않는다.** |
| **설명** | 흐름: 지오코딩 → 구역·최근접 대여소 → **도보 소요** → `rent_time = ref + walk` → ML 가용 예측. 목적지 도착 시각 역산, “집→대여소→목적지” 전체 소요는 없다. |
| **연관 한계** | **문제 정식화 한계** ? 과제 제목의 “언제 출발해야 **탈 수 있을까**”를 **“할당 시간 안에 대여소에 갈 수 있고, 그때 자전거가 있을 것 같다”**로 근사했다. 목적지 일정과 연동된 출발 권장은 **현 버전 범위 밖**이다. |
| **영향** | 사용자가 “9시까지 광화문”을 입력할 수 없고, **이동 목적·자전거 구간**이 결과에 반영되지 않는다. |

### 6-2. 과거 패턴으로 “지금” 시각 예측

| 항목 | 내용 |
|------|------|
| **이슈** | ML은 2024년 3개월로 학습했으나, Notebook §6 기본값은 `datetime.now()` 기준 날짜·시각이다. |
| **설명** | `_features_for`는 `rent_time`의 month·season·hour만 쓰며, **연도는 특성에 없다.** 2026년 실행 시에도 2024년과 같은 “4월 화요일 8시” 패턴을 적용한다. |
| **연관 한계** | **시간 일반화 한계** ? 코로나 이후 수요 변화, 대여소 증감, 정책 변경은 반영되지 않는 **과거 스냅샷 기반 추정**이다. |
| **영향** | 데모는 `DEMO_REF_TIME = 2024-04-15`가 학습 분포와 맞지만, 실사용 날짜는 **분포 외 extrapolation**에 가깝다. |

### 6-3. API vs Demo 소요시간

| 항목 | 내용 |
|------|------|
| **이슈** | Kakao Mobility 실패 시 `haversine` 직선거리 ÷ 고정 속도(도보 4.5km/h, 자전거 15km/h)로 대체한다. REST API IP 미등록 시 지오코딩도 좌표 Demo로 대체한다. |
| **설명** | `_mobility_duration`은 mode를 구분하지 않고 경로 API 결과만 쓴다. Demo는 곡률·신호·일방통행을 무시한다. API 성공 시에도 출발지가 대여소와 매우 가까우면 **도보 0분**처럼 보일 수 있다. |
| **연관 한계** | **이동 시간 추정 한계** ? `timing_feasible`, `depart_home`, `rent_time`이 **실제 도로 환경과 어긋날** 수 있다. 가용성 ML 오차와 별개로 **일정 역산 자체가 부정확**해진다. |
| **영향** | API/Demo 모드에 따라 **같은 주소도 다른 판정**이 나올 수 있으며, Notebook은 모드를 출력하지만 불확실성 정량화는 하지 않는다. |

### 6-4. 모델 간 예측 불일치와 단일 모델 판정

| 항목 | 내용 |
|------|------|

## 9. 결론: 한계를 어떻게 읽을 것인가

본 프로젝트 구현은 **공공데이터 ETL → 공간 군집 → 시·공간 회귀 → 지도 API 보조 → Notebook 데모**까지의 **오프라인 분석 파이프라인**으로서 3학년 데이터마이닝 과제 요건(군집화, 선형 회귀, Random Forest, 하이퍼파라미터 튜닝, 비교)을 충족한다.

다만 다음을 전제로 결과를 해석해야 한다.

1. **예측은 실시간 따릉이 API가 아니라**, 2024년 일부 월의 **과거 시간대별 패턴**이다.  
2. **공간 단위는 구역 평균**이며, 사용자가 쓰는 **한 대여소**와 1:1이 아니다.  
3. **출발 권장은 “목적지 역산”이 아니라**, “기준 시각부터 할당 시간 내 도보로 대여소 도달 + 그때 가용할 것으로 기대”에 가깝다.  
4. **선형 회귀 성능 저하는 모델 고장이라기보다**, 범주형 `cluster_id` 처리와 문제 난이도가 겹친 **구조적 한계**로 보는 것이 맞다.  
5. **RF Tuned를 판정에 쓰는 것**은 테스트 지표상 합리적이나, 모델 간 편차가 큰 입력에서는 **불확실성이 크다**는 점을 함께 보고해야 한다.

이 전제들을 명시한 상태에서 Notebook 결과는 **“가능성 탐색·교육용 의사결정 보조”**로 적절하며, **실제 출발 시각의 단정적 권고**로 쓰기에는 §2~§6의 한계가 누적된다.
